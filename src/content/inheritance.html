The Deep Hierarchy

Inheritance: Hierarchy, State, and the Fragile Base

Inheritance represents the IS-A relationship. While powerful, it is the most dangerous tool in OOP due to tight coupling. We explore the memory layout, the dangers of deep hierarchies, and the modern transition to Composition and Sealed Classes.

1. IS-A vs. HAS-A (The Golden Rule)

Use inheritance only if you can say "A [Subclass] IS-A [Superclass]." However, even when true, you must ask if it's the right abstraction.

  • CreditCardProcessor IS-A PaymentProcessor (Inheritance).
  • PaymentService HAS-A PaymentProcessor (Composition).
🤔 Why This Exists
Inheritance exists because large systems repeat behavior. Without it, you would copy-paste the "Connect to Payment Gateway" code 50 times. But inheritance also creates tight coupling, which is why modern systems often prefer composition to share behavior.

Example: Enterprise Payment Architecture

public abstract class PaymentProcessor {
    protected String transactionId;

    public PaymentProcessor(String txId) { 
        this.transactionId = txId; 
    }

    // Concrete behavior shared by all subclasses
    public void logAudit() {
        System.out.println("[AUDIT] Processing: " + transactionId);
    }

    // Abstract behavior enforced on subclasses
    public abstract void processPayment(double amount);
}

public class StripeProcessor extends PaymentProcessor {
    private String apiKey;

    public StripeProcessor(String txId, String apiKey) {
        super(txId); // MUST call parent constructor
        this.apiKey = apiKey;
    }

    @Override
    public void processPayment(double amount) {
        logAudit(); // Inherited method
        System.out.println("Calling Stripe API with key... Amount: $" + amount);
    }
}

public class CryptoProcessor extends PaymentProcessor {
    public CryptoProcessor(String txId) { super(txId); }

    @Override
    public void processPayment(double amount) {
        logAudit();
        System.out.println("Validating blockchain transaction... Amount: " + amount + " ETH");
    }
}

// Usage — Polymorphism in action
PaymentProcessor p1 = new StripeProcessor("TX-991", "sk_live_123");
p1.processPayment(150.00); // Triggers Stripe implementation

PaymentProcessor p2 = new CryptoProcessor("TX-992");
p2.processPayment(0.05); // Triggers Crypto implementation

2. Memory Layout: The Layered Object

When you create a Child object, the JVM creates a unified memory block with all fields from the entire hierarchy.

[ CHILD OBJECT ON HEAP ] ┌───────────────────────────┐ │ Object Header │ ├───────────────────────────┤ │ Parent Fields (hidden) │ <── Still takes space! ├───────────────────────────┤ │ Child Fields │ └───────────────────────────┘
🕵️ Senior Insight: The Memory Cost
Even if you "shadow" a field (Parent and Child both have int id), both fields exist in memory. The JVM simply chooses which one to show you based on the reference type.

3. The "super" Keyword Deep Dive

public class Employee {
    protected String name;
    protected double baseSalary;

    public Employee(String name, double baseSalary) {
        this.name = name;
        this.baseSalary = baseSalary;
    }

    public double calculatePay() {
        return baseSalary;
    }

    public String toString() {
        return name + ": $" + calculatePay();
    }
}

public class Manager extends Employee {
    private double bonus;

    public Manager(String name, double base, double bonus) {
        super(name, base); // call parent constructor
        this.bonus = bonus;
    }

    @Override
    public double calculatePay() {
        return super.calculatePay() + bonus; // extend parent behavior
    }
}

Employee mgr = new Manager("Alice", 5000, 2000);
System.out.println(mgr); // Alice: $7000.0

4. The "Diamond Problem" (The Bytecode Fix)

Java forbids multiple inheritance of Classes but allows it for Interfaces with default methods.

interface Flyable {
    default void move() { System.out.println("Flying"); }
}

interface Swimmable {
    default void move() { System.out.println("Swimming"); }
}

// Diamond! Must resolve manually
class Duck implements Flyable, Swimmable {
    @Override
    public void move() {
        Flyable.super.move();   // explicitly choose
        Swimmable.super.move();
    }
}

Duck d = new Duck();
d.move(); // prints both "Flying" and "Swimming"

5. The "Fragile Base Class" Problem (Design Smell)

🤢 Design Smell: Deep Inheritance Tree
If your class hierarchy is Object -> Entity -> User -> AdminUser -> SuperAdminUser, you have a Fragile Base Class. Changing one method in Entity will inadvertently break SuperAdminUser.
🧠 How Senior Engineers Think
"Prefer Composition over Inheritance." Juniors use inheritance for code reuse. Seniors use it ONLY for polymorphic behavior substitution. If you just want to reuse a database connection, inject it (Composition). Don't inherit from BaseDatabaseEntity.

6. Bad Code → Good Code: The Composition Shift

Let's fix a classic architectural mistake.

❌ Bad: Inheritance (Rigid)
class Notification { 
    void send() { ... } 
}
class EmailNotification extends Notification { ... }
class SMSNotification extends Notification { ... }

// Requirements change: User wants BOTH Email and SMS!
// Now what? EmailAndSMSNotification? (Explosion of classes)
✅ Good: Composition (Flexible)
interface MessageChannel {
    void deliver();
}
class EmailChannel implements MessageChannel { ... }
class SMSChannel implements MessageChannel { ... }

class NotificationService {
    // HAS-A relationship. We can inject a List of channels!
    List<MessageChannel> channels;
    
    void sendAll() {
        channels.forEach(MessageChannel::deliver);
    }
}
⚙️ LLD Connection
Composition enables the Strategy Pattern. By injecting MessageChannel dependencies at runtime (via Spring or manually), you achieve Loose Coupling. You can swap out an SMS provider without touching the NotificationService class.

7. Sealed Classes (Java 17)

public sealed class Shape
    permits Circle, Rectangle, Triangle {

    public abstract double area();
}

public final class Circle extends Shape {
    private final double radius;
    public Circle(double r) { this.radius = r; }
    @Override
    public double area() { return Math.PI * radius * radius; }
}

public final class Rectangle extends Shape {
    private final double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }
    @Override
    public double area() { return w * h; }
}

// Exhaustive switch (Java 21)
double describeArea(Shape s) {
    return switch (s) {
        case Circle c    -> c.area();
        case Rectangle r -> r.area();
        case Triangle t  -> t.area();
    }; // no default needed — sealed!
}

Example: Field Shadowing Trap

class Parent {
    String name = "Parent";
}

class Child extends Parent {
    String name = "Child"; // SHADOWS parent field
}

Parent obj = new Child();
System.out.println(obj.name); // "Parent" — resolved by REFERENCE type!

Child obj2 = new Child();
System.out.println(obj2.name); // "Child"
Key Takeaways — Section 04C
  • Inheritance is White-Box reuse (internal details leak to children).
  • Always use @Override to catch compile-time errors.
  • Favor Composition: "Has-A" is usually safer than "Is-A."
  • Shallow is better: Avoid the "God Parent" class.
  • Sealed Classes give you controlled inheritance for domain modeling.

Interview Questions — Section 04C

What is the 'Object' class's role in the hierarchy?▶
Every class implicitly inherits from java.lang.Object. It provides equals(), hashCode(), toString(), and wait/notify. It is the root of all polymorphism in Java.
Why doesn't Java support multiple inheritance for classes?▶
To avoid the Diamond Problem. If Class A and B have a field x, and Class C inherits from both, which x does it get? Java avoids this ambiguity. Interfaces with default methods re-introduced a controlled version.
What is the difference between 'extends' and 'implements'?▶
extends creates an IS-A relationship with a class (single inheritance). implements creates a CAN-DO relationship with an interface (multiple allowed). A class can extend one class and implement many interfaces simultaneously.
What happens if a child class doesn't call super() in its constructor?▶
The compiler automatically inserts a call to the parent's no-args constructor (super()) as the first line of the child constructor. If the parent class does not have a no-args constructor, the code will fail to compile unless the child explicitly calls a matching super(...) with arguments.
